# Layer 3 — Orchestration: Тактика, Оперативність, Стратегія

> Цей документ описує систему оркестрації — як 9 атомарних операцій, 3 рівні горизонту і 4 контури перетворюють прості сигнали користувача на живу, що дихає, стрічку. Ключовий принцип: **AI генерує значення, код контролює ритм.**

---

## 9 операцій оркестрації

Не треба хардкодити сто типів гумору. Семантичний зміст ниток формулює AI у природній мові. **Жорстко заданими** мають бути лише операції:

| Операція | Що робить | Коли використовувати |
|----------|-----------|---------------------|
| `probe` | Перевірити нову можливість | Cold start, exploration |
| `deepen` | Піти глибше в знайдений нерв | Після підтвердження нитки |
| `mutate` | Залишити нерв, змінити оператор | Коли форма приїлася, нерв живий |
| `transfer` | Перенести механізм у інший світ | Перевірка глибини vs. контексту |
| `bridge` | Схрестити дві нитки | Коли дві нитки мають прихований зв'язок |
| `contrast` | Змінити емоційне паливо | Від агресії до ніжності і навпаки |
| `callback` | Повернути старий мотив після паузи | Shared history, inside joke |
| `rest` | Дати нитці відпочити | Fatigue високий |
| `wildcard` | Контрольований стрибок за межі профілю | Novelty debt, profile expansion |

### Приклад мислення системи

```
НЕ: "Дай ще один офісний жарт"

А: "Нитка гаряча, але два останні попадання вже підвищили втому.
    Не повторювати її прямо.
    Спробувати transfer: той самий механізм формалізації людського
    хаосу, але у психотерапії.
    Після цього дати wildcard."
```

## 3 рівні оркестрації

### Тактичний рівень: остання реакція

**Горизонт:** наступні 1–3 картки

Відповідає на:
- чи продовжувати поточну нитку
- чи змінити форму
- чи треба відступити
- чи сильний сигнал відкрив нову гілку
- чи готова картка стала неактуальною

> Це переважно **прості правила + невеликий state update**. Без GPT.

### Оперативний рівень: ритм сесії

**Горизонт:** поточна сесія

Бачить не окремий жарт, а **послідовність**:

```
попадання → легке відхилення → новий probe → сильне попадання
→ зміна температури → wildcard → callback
```

Контролює:
- повторення
- емоційний тембр
- складність
- частоту ризику
- кількість знайомого й нового
- втому від голосів і форматів

> Саме цей рівень створює відчуття: "Я не знаю, що буде далі, але довіряю, що це може бути добре."

### Стратегічний рівень: спільна історія

**Горизонт:** дні й місяці

Вирішує:
- які нитки справді стійкі
- які були випадковими
- де з'явився глибший спільний механізм
- які нитки можна об'єднати
- яку стару нитку варто повернути
- що користувач уже переріс
- які домени стали достатньо контекстними для вузького гумору

> Стратегічний рівень забезпечує **long tail** — не через нескінченну генерацію, а через накопичення спільної семантичної історії.

## 4 контури системи

```
┌─────────────────────────────────────────────────────┐
│                   USER SEES CARD                     │
│                        ↓                             │
│              EVENT IS RECORDED                       │
│                        ↓                             │
│         DETERMINISTIC TACTICAL UPDATE                │
│                        ↓                             │
│      NEXT READY CARD RETURNED IMMEDIATELY            │
│                        ↓                             │
│          IF NEEDED → REFLECTION JOB                  │
│                        ↓                             │
│       THREADS / SESSION PLAN PATCHED                 │
│                        ↓                             │
│              COMPOSITION JOB                         │
│                        ↓                             │
│       NEW CANDIDATES ENTER FRONTIER                  │
│                        ↓                             │
│   USER SEES CONSEQUENCES OF THEIR SIGNALS            │
└─────────────────────────────────────────────────────┘
```

### Контур 1: Fast Feed Loop (мілісекунди)

```
show → react → record → select next
```

**Без GPT.** Наступні 2–3 картки вже в буфері. Клієнт префетчить. `next-card` вибирає найдоречнішу готову можливість.

### Контур 2: Tactical Loop (мілісекунди)

Оновлює:
- `heat`, `fatigue`
- session tempo
- novelty debt
- recent move history
- refill necessity

**Також без GPT.**

### Контур 3: Semantic Reflection Loop (секунди, у background)

Періодично питає AI: "Що ми насправді дізналися з останніх реакцій?"

Він змінює нитки.

**Тригери:**
```
heart (кожен heart — сильний сигнал, бо він рідкісніший за swipe)
share
два heart серед останніх п'яти карток
6–8 карток після останнього reflection
суперечність між очікуваним і фактичним
session end
frontier став inconsistent з evidence
```

### Контур 4: Strategic Long-Memory Loop (раз на кілька сесій)

AI переглядає не картки, а **історію розвитку ниток**:
- які нитки були випадковими / стали стійкими
- які є проявами одного глибшого механізму
- що перенасичене / що давно не поверталося
- де достатньо shared history для callback
- який новий домен варто відкрити

**Тригери:**
```
3 завершені сесії
30–50 показаних карток
5 нових canon cards
кілька ниток із семантичною близькістю
повернення після довгої перерви
```

## Rolling Frontier — живий мікробуфер

> [!CAUTION]
> Не генерувати 30–50 текстів наперед. Це перетворює живий процес на заздалегідь надруковану газету.

### Структура фронтиру

```
slot 1 — anchor / likely hit (deepen)
slot 2 — nearby mutation
slot 3 — transfer or contrast
slot 4 — wildcard
slot 5 — callback candidate (optional)
slot 6 — replacement candidate (optional)
```

- 2–4 ready cards
- 2–3 speculative candidates
- все прив'язане до `state_version` / `thread_version`
- після сильного сигналу частина кандидатів стає stale

> Користувачеві не показують у зафіксованому порядку. Після кожного значущого сигналу оркестратор **заново** вирішує, яка з готових можливостей тепер правильна.

## Hard Constraints — механічні інваріанти

> AI добре розуміє семантичні відтінки. Але він **погано** тримає довгострокову дисципліну.

Тому застосунок **жорстко** контролює:

```
✗ не показувати одну нитку більше 2 разів близько підряд
✗ не повторювати той самий формат у 3 з 5 карток
✗ після сильного hit не давати одразу копію
✗ після save запланувати callback, але не зараз
✗ після fast skip не вбивати нитку
✗ повертати callback лише після достатньої паузи
✗ не використовувати personal fact у quality fund

✓ після сильного hit збільшити і heat, і fatigue
✓ тримати хоча б один probe/wildcard у короткому горизонті
✓ погашати novelty debt
✓ після серії слабких сигналів змінити не лише тему, а оператор або голос
```

### Поділ влади

```
Механіка вирішує:               AI вирішує:
├── Зараз потрібен transfer      ├── Як перетворити цей нерв
├── Температура має бути нижчою  │   у текст, що не є повтором
├── Не використовувати 2 останні │
│   голоси                       └── Який семантичний experiment
├── Нитка A гаряча, перенасичена     зробити в рамках місії
└── Нитка B давно не поверталася
```

## Нитка — це гіпотеза, не істина

> [!CAUTION]
> Найбільший ризик: AI один раз вирішить "Цій людині смішна корпоративна безпорадність" і почне **генерувати докази власної теорії**. Так народжується не глибина, а самозамкнений фанфік.

Кожна нитка зберігає:

```json
{
  "hypothesis": "формальна система випадково оголює людську безпорадність",
  "confidence": 0.64,
  "positive_evidence": ["card_17", "card_31"],
  "counter_evidence": ["card_42"],
  "open_question": "спрацьовує механізм формалізації живого чи лише знайомий контекст?",
  "heat": 0.78,
  "fatigue": 0.31,
  "depth": 3,
  "status": "active"
}
```

Система повинна вміти сказати:
- "ми це **підозрюємо**"
- "ми це підтвердили **в одному контексті**"
- "перенесення **не спрацювало**"
- "можливо, ми неправильно **зрозуміли причину**"
- "цю нитку треба **розділити** на дві"

### heat vs. confidence vs. fatigue

| Параметр | Що вимірює | Після heart |
|----------|-----------|-------------|
| `heat` | Наскільки нитка зараз здатна дати результат | ↑ зростає |
| `confidence` | Наскільки ми розуміємо, **чому** вона працює | НЕ автоматично |
| `fatigue` | Наскільки небезпечно її повторювати | ↑ зростає |

> Один hit не означає "ми зрозуміли людину". Він означає: **тут щось є, треба розкопати**. Це епістемічна скромність системи.

## Як одна нитка реально розвивається

Користувач засміявся з:

> На daily сказав, що заблокований.  
> Не став уточнювати, що як особистість.

Система **НЕ** записує: `likes: Java humor`.

Вона створює слабку гіпотезу:
> знайома професійна форма випадково оголює приватну людську правду

Потім показує **не три daily-жарти**, а:

| Тип | Текст | Що тестує |
|-----|-------|-----------|
| **Deepen** | "У self-assessment попросили описати зони розвитку. Я не знав, що поле підтримує роман." | Чи працює deeper professional context |
| **Transfer** | "На терапії запитали, що я зараз відчуваю. Сказав, що повернуся з уточненням після синку із собою." | Чи механізм ширший за IT |
| **Contrast** | "Ми всі вдаємо, що знаємо, як жити. Просто деякі вже красиво оформили документацію." | Інше emotional fuel |
| **Probe** | Абсолютно інший механізм — ніжна людська буквальність | Profile expansion |

Якщо transfer у терапію теж спрацював → нерв ширший за IT.
Якщо IT deepen працює, а transfer ні → сила у знайомому ритуалі.

> **Оце і є майнинг: не продовжувати успішний текст, а розрізняти можливі причини його успіху.**

## System Quality Dashboard — "МРТ" алгоритму для конкретного user_id

> [!IMPORTANT]
> Це **не** продуктові метрики (retention, DAU). Це інструмент верифікації: чи алгоритм, який ми задумали, поводиться як **живий організм** — чи він скотився в тупий рекомендаторний цикл. Один дашборд, один `user_id`, п'ять чисел + один графік.

### Парето-принцип: п'ять чисел, що покажуть усе

Усі ці числа видобуваються з **існуючих таблиць** (`threads`, `cards`, `events`, `sessions`). Жодної нової сутності.

| # | Метрика | SQL-джерело | Що показує | Тривожний сигнал |
|---|---------|-------------|------------|------------------|
| 1 | **Resonance Rate** | `events` heart+share / `cards` shown | Загальна якість попадань | < 15% після 50 карток |
| 2 | **Transfer Success** | `cards` з move=transfer,bridge + heart / усі transfer,bridge shown | Чи AI **реально розуміє** глибинний механізм, а не поверхню | = 0 після 30 карток |
| 3 | **Thread Diversity** | COUNT DISTINCT `thread_id` серед останніх 20 shown cards | Чи система не застрягла на одній нитці | < 2 нитки в 20 картках |
| 4 | **Breathing Index** | Кількість ниток із status `resting` або `dormant` / усі `active`+`resting` | Чи оркестратор реально відпускає нитки відпочивати | = 0 після 3 сесій |
| 5 | **Depth Progression** | AVG(`depth`) active threads зараз vs. AVG(`depth`) тиждень тому | Чи нитки реально заглиблюються з часом | depth не зросла за 5 сесій |

### SQL View (одним запитом для user_id)

```sql
CREATE VIEW user_algorithm_health AS
SELECT
  u.user_id,

  -- 1. Resonance Rate
  ROUND(
    COUNT(e.id) FILTER (WHERE e.event_type IN ('heart','share'))::numeric
    / NULLIF(COUNT(DISTINCT c_shown.id), 0), 2
  ) AS resonance_rate,

  -- 2. Transfer Success Rate
  ROUND(
    COUNT(e.id) FILTER (
      WHERE e.event_type = 'heart'
        AND c_all.move IN ('transfer','bridge')
    )::numeric
    / NULLIF(COUNT(DISTINCT c_all.id) FILTER (
        WHERE c_all.move IN ('transfer','bridge') AND c_all.status = 'shown'
      ), 0), 2
  ) AS transfer_success_rate,

  -- 3. Thread Diversity (останні 20 shown)
  (SELECT COUNT(DISTINCT unnest(source_thread_ids))
   FROM cards
   WHERE user_id = u.user_id AND status IN ('shown','hearted')
   ORDER BY shown_at DESC LIMIT 20
  ) AS recent_thread_diversity,

  -- 4. Breathing Index
  ROUND(
    COUNT(t.id) FILTER (WHERE t.status IN ('resting','dormant'))::numeric
    / NULLIF(COUNT(t.id) FILTER (WHERE t.status IN ('active','resting')), 0), 2
  ) AS breathing_index,

  -- 5. Avg active thread depth
  ROUND(AVG(t.depth) FILTER (WHERE t.status = 'active'), 1) AS avg_depth

FROM user_minds u
LEFT JOIN threads t ON t.user_id = u.user_id
LEFT JOIN cards c_shown ON c_shown.user_id = u.user_id AND c_shown.status IN ('shown','hearted')
LEFT JOIN cards c_all ON c_all.user_id = u.user_id
LEFT JOIN events e ON e.user_id = u.user_id AND e.card_id = c_shown.id
GROUP BY u.user_id;
```

> [!NOTE]
> Цей view — **діагностичний**, не для юзера. Ми дивимося на нього самі. На MVP це `SELECT * FROM user_algorithm_health WHERE user_id = '...'`. Пізніше — Grafana чи будь-який простий дашборд.

### Єдиний графік: Thread Breathing Timeline

```
Thread A:  ███▓▓░░░░░░░▓▓████░░░░▓▓▓██████
Thread B:  ░░░░░░████▓▓░░░░░░░░░░████▓▓▓░░
Thread C:  ░░░░░░░░░░░░░░████▓▓▓▓░░░░░░░░░
           ─────────────────────────────────→ time (cards shown)

█ = active (used in shown card)
▓ = resting (exists but not used)  
░ = not yet created / dormant
```

**Як читати:** Здоровий алгоритм показує **хвилі** — нитки спалахують, гаснуть, повертаються. Якщо одна нитка — суцільна лінія `████` без пауз — оркестратор не працює, ми тупо повторюємо одне й те саме. Якщо всі нитки паралельно активні без дихання — ми не будуємо глибину, а розбризкуємо.

### Що робити з результатами

| Діагноз | Що означає | Де фіксити |
|---------|-----------|-------------|
| Resonance Rate низький | AI генерує слабкий контент або неправильно зчитує нерви | Промпти в `ai-worker` |
| Transfer Success = 0 | AI клепає фейкові "переноси", механізм не зрозумілий | Промпт `reflect` + якість `open_question` в нитках |
| Thread Diversity = 1 | Оркестратор не відпускає гарячу нитку | Hard constraints у `next-card` |
| Breathing = 0 | Fatigue-логіка не працює, нитки ніколи не відпочивають | Tactical loop у `record-signal` |
| Depth не росте | Система не заглиблюється, застрягла на поверхні | Промпт `compose` + trigger `strategic_reflect` |
