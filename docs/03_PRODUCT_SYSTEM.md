# Layer 2 — Product System: Flywheels, Lifecycle & Signals

> Цей документ описує продуктову механіку — як flywheel-логіка перетворює прості жести користувача на глибинне розуміння, і чому продукт не вигорає з часом.

---

## Три Flywheel, які працюють разом

### 1. Особистий Flywheel (Personal Depth)

```
краще попадання
  → чистіший сигнал
    → глибша нитка
      → більш конкретний контекст
        → ще сильніше попадання
```

Це flywheel **глибини** — чим точніше система попадає, тим якісніший фідбек отримує, тим точніше стає.

### 2. Flywheel новизни (Novelty Expansion)

```
знайдений механізм
  → transfer / mutation / bridge
    → новий домен резонансу
      → ширша карта людини
        → менша залежність від одного патерну
          → довший життєвий цикл продукту
```

Це flywheel **ширини** — система не замикається на одному нерві, а переносить знайдені механізми в нові домени.

### 3. Колективний Flywheel (Quality Fund)

```
сильні особисті попадання
  → очищені semantic recipes
    → кращий global quality fund
      → кращий cold start для нових людей
        → швидше перше попадання
          → більше якісних сигналів
```

Це flywheel **масштабу** — кожне успішне попадання одного користувача покращує першу зустріч для інших.

> [!IMPORTANT]
> Все запускається одним простим жестом: **скрол → ❤️ heart → ↗️ share**.

## Від Onboarding до Lifecycle

### Onboarding ≠ анкета

**НЕ починати банально з:** опитувань жанрів ("dark humor чи dad jokes", "які теми любиш").

> [!TIP]
> **Хоч це і корисні дані!**
> Вік, країна та мова — це справді потрібний фундамент. Вони дають культурний код, prior-контекст і мову генерації. Але вони **не відкривають саму Laugh Topology**. Ми збираємо демографію для "розуміння світу", але реальну карту резонансу будуємо виключно через емпіричні реакції.

### Три типи інформації з onboarding

| Тип | Що знімаємо | Приклади |
|-----|-------------|---------|
| **Світ** | Які реальності знайомі | робота, міграція, стосунки, сім'я, батьківство, підприємництво |
| **Дозвіл** | Куди можна заходити | мат, секс, смерть, релігія, політика, жорстокість |
| **Резонанс** | Які шляхи у сміх працюють | серія ортогональних textual probes |

### Feed І Є продовження onboarding

> [!TIP]
> Onboarding не закінчується перед feed. Користувач не бачить моменту "калібрування завершене". Він просто помічає, що попадання стають частішими.

### Перші 8–12 карток — діагностичні probes

Не випадкові жарти й не представники жанрів. **Ортогональні проби:**

```
одна тема — різні механізми
один механізм — різні теми
одна напруга — різна психологічна дистанція
одна структура — різний голос
миттєвий удар vs. slow burn
впізнавання vs. абсурд
ніжність vs. агресія
```

Після першого сильного сигналу система відкриває дві гілки:
1. Перевірити, що **саме** спрацювало
2. Дати щось достатньо **інше**, щоб не замкнути профіль

## Сигнали — дві явні дії + неявні

> [!IMPORTANT]
> **MVP-принцип:** мінімум когнітивного навантаження, максимум якості сигналу. Дві кнопки — ❤️ і ↗️. Решта — неявні сигнали, які клієнт збирає автоматично.

### Signal Semantics

| Сигнал | Тип | Що означає | Що робить у системі |
|--------|-----|-----------|---------------------|
| **❤️ Heart** | explicit | "Це спрацювало / це мене пробило" | Позитивний сигнал для нитки **+** автоматичне збереження в Personal Canon |
| **↗️ Share** | explicit | "Цей текст виконує соціальну функцію" | Окрема вісь. **Не** автоматично сильніший за heart |
| **Dwell / stop-scroll** | implicit | Увага, але не обов'язково позитив | Впливає на впевненість лише **разом** із подальшою поведінкою |
| **Fast skip** | implicit | Слабкий negative | Не вбиває нитку. Серія fast skip на споріднених рецептах — вже доказ |
| **Back / reread** | implicit | Slow-burn, cognitive residue | Текст працює відкладено |

> [!TIP]
> **Чому laugh + save = heart?** Сучасна людина не хоче ухвалювати два рішення ("це смішно" + "а чи зберегти?"). Один тап — і все. Якщо короткий текст реально пробив — ймовірність, що людина хоче його зберегти, дуже висока. А якщо Personal Canon стане "зашумленим" — це розкішна проблема, яку вирішимо пізніше, додавши ⭐ для суперхітів.

### Signal Vector (а не скаляр)

```json
{
  "attention": 0.87,
  "mirth": 1.0,
  "identity_resonance": 0.25,
  "social_utility": 0.0,
  "rejection": 0.0,
  "slow_burn_probability": 0.15
}
```

> Dwell нормалізується за довжиною тексту. 3 секунди для 7 слів і 3 секунди для 100 слів — різні події.

## Personal Canon

> [!IMPORTANT]
> Personal Canon — це **не** "saved posts". Це зростаюча колекція текстів, які реально пробили людину.

Коли людина натискає ❤️, текст автоматично потрапляє в канон. Вона каже:
- "це мене побачило"
- "це сформулювало мою правду"
- "це смішно саме моїм способом"
- "це я хочу перечитати"

### Три функції Personal Canon

| Функція | Як працює |
|---------|-----------|
| **Цінність для користувача** | Колекція текстів, які реально щось відкрили. Людина може переглядати й видаляти зайве |
| **Пам'ять для AI** | Найчистіші exemplars голосу й резонансу |
| **Матеріал для callback** | Система продовжує не текст, а знайдений механізм |

> НЕ показувати діаграму "Ти на 73% любиш екзистенційний deadpan". Нехай людина пізнає себе через те, що накопичується в каноні.

> [!NOTE]
> **Еволюція:** Коли канон виросте до 50+ текстів, ми можемо додати ⭐ (star) для виділення суперхітів, не ламаючи наявний UX.

## Quality Fund для інших

Кожне сильне попадання може породити два активи:

### 1. Reusable Exact Card
Текст достатньо універсальний, не містить приватного контексту → показати іншим.

### 2. Semantic Recipe
Текст персональний, але **механізм** переносний:

```
charged_reality: ритуал професійної компетентності
transformation:  офіційна мова випадково зізнається в безпорадності
voice:           dry participant
distance:        self-inclusive, non-hostile
payoff:          [recognition, relief]
failure_modes:   [generic burnout joke, obvious tech vocabulary]
```

### 3. Diagnostic Value
Картка, що добре розділяє дві гіпотези (абсурд чи semantic reversal? агресію чи relief?). Може бути ціннішою для cold start, ніж найвіральніший жарт.

> Quality Fund — це **не leaderboard** найсмішніших текстів. Це бібліотека сильних текстів + сильних рецептів + сильних діагностичних тестів.

## Anti-Burnout закони

| Закон | Механіка |
|-------|----------|
| **Hit відкриває гілку, не серію копій** | Після попадання досліджується причина, не повторюється оболонка |
| **Попадання підвищує і heat, і fatigue** | Що ціннішою стає нитка — то обережніше її використовувати |
| **Після кількох близьких попадань — rest** | Нитка не зникає, а дозріває для callback |
| **Новизна різної дистанції** | Маленька мутація, перенесення, синтез, повний wildcard |
| **Старі нитки повертаються глибше** | Більше контексту, точніший голос, перехрещення з новими нервами |
| **Не кожен текст має бути найсмішнішим** | Деякі картки змінюють температуру, розширюють карту, створюють очікування |

> Продукт оптимізує **не heart rate кожної картки**, а **бажання побачити наступну**.

## North-Star метрики

| Метрика | Що вимірює |
|---------|-----------|
| **Resonance rate** | Частка карток із сильним сигналом |
| **Diversity-adjusted resonance** | Попадання на нових операторах/transfer/bridge, не повтори старого |
| **Canon growth** | Як часто користувач зберігає тексти |
| **Session continuation** | Чи хоче побачити наступну |
| **Return depth** | Повернення через день/тиждень/місяць |
| **Thread expansion** | Чи відкриваються нові стійкі нитки |
| **Transfer success** | Чи працює механізм поза вихідною темою |

### North-Star Formula

> **Частота сильних несподіваних попадань без семантичного виснаження.**

## Як виникає справжня глибина (6 рівнів)

```
Рівень 1: Широкий світ           → робота в IT
Рівень 2: Конкретний контекст     → Java backend
Рівень 3: Повторювані ситуації    → Spring context, daily, code review, production incident
Рівень 4: Людська суперечність    → мовою контролю, поки система оголює колективну фантазію
Рівень 5: Перенесення нерву       → та сама суперечність у стосунках, медицині, дорослому житті
Рівень 6: Callback & shared history → inside joke з додатком, спільна мова
```

> [!TIP]
> **LTV** народжується не з infinite content, а з **накопичення спільної мови** між людиною і системою. Це і є справжній switching cost: інший застосунок може знати, що ти програміст; цей пам'ятає, яким способом ви разом навчилися сміятися з твоєї реальності.

---

## Long Tail із трьох джерел

1. **Контекстна деталізація** — робота → backend → production incident, code review, Spring context
2. **Семантичне узагальнення** — не production incident, а "ритуал компетентності перед обличчям систем, якими ніхто не керує" → тепер переносимо
3. **Shared language** — callback, що зрозумілий завдяки спільній історії. Не повторити жарт, а повернути його внутрішню логіку в новій ситуації
